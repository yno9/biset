# biset Architecture Notes

## Build flow

biset is distributed as a **single HTML file**. `dist/app.js` is inlined into `dist/index.html` at build time, so the app runs directly via `file://` with no dev server required (local-first design).

### Build and deploy

```sh
bun run build
```

This runs the `build` script in `package.json`:
```
bun build src/main.ts --outfile=dist/app.js --target=browser --minify-syntax --minify-whitespace --define ... && bun run scripts/inline.mjs
```

`scripts/inline.mjs` reads `src/style.css` and `dist/app.js` and splices them into `dist/index.html` replacing the placeholder comments. Running `bun build` alone updates `dist/app.js` but leaves `dist/index.html` with the old inlined code ŌĆö always use `bun run build`.

**Deploy** to production (`v1`) with `./deploy.sh [app|landing|all]`. Two **separate artifacts** are served from the same jmapap host ŌĆö never conflate them:

| Artifact | Source | Server dir | Served at | Config |
|---|---|---|---|---|
| **app** (chat client) | `dist/index.html` (built) | `/root/jmapap/app/` | `t.biset.md` | jmapap `web_root:"app"` |
| **landing** (public site) | `home/*` (index + assets + `tos.html`) | `/root/jmapap/apex/` | `biset.md` apex | Caddy `root * /root/jmapap/apex` |

`deploy.sh` builds (app only), uploads, then verifies via **sha match + live URL** so a wrong-directory or stale deploy fails loudly. Do not `scp` by hand ŌĆö deploying the app over the landing dir (or vice versa) silently breaks one of the two sites, and file mtimes alone won't reveal it.

## Layout

Two-pane app (`#app`):
- **`#left-pane`** ŌĆö card with margin, collapsible. Contains search, inbox list, LP_COMMANDS.
- **`#right-col`** ŌĆö flex column containing `#outer` (scrollable) and `#reply-dock` (sticky bottom).
  - `#outer` contains `#thread-title-row` (sticky top), `#right-pane` (thread/cmd content).

### Mobile (`Ōēż720px`)

Left pane is hidden by default. `#app.lp-enabled.show-left` shows the left pane full-screen and hides `#right-col`. `#right-pane` gets `width: 100%` to override the desktop calc.

### Single-col mode

`#app.single-col.lp-enabled` collapses the left pane to `width: 0` via CSS transition (shrinks from the right). Triggered by Cmd+B or the toggle button.

## Routing

Hash-based:
- `#new` ŌåÆ new-user onboarding page (no app chrome, full-screen)
- `#account`, `#config`, `#compose`, `#debug` ŌåÆ LP_COMMANDS pages inside the normal app shell (the complete, fixed set ŌĆö `main.ts`'s `MENU_PAGE_NAMES`)
- `#contact` ŌåÆ opens a specific inbox thread ŌĆö see below
- `#compose/<addr>` ŌåÆ compose page with To pre-filled

`init()` in `main.ts` reads `location.hash` on load and dispatches accordingly. After account creation on the `#new` page, `showApp()` + `setupLeftPane()` + `startPolling()` run before navigating to `#account`.

**A conversation permalink is just the contact (2026-07-14)** ŌĆö `utils.ts`'s `inboxToHash`/`parseInboxHash`. Which of the user's own logged-in identities/mailboxes a conversation happens to live under is a self-referential detail with no business being in a URL that's fundamentally "who this is with" ŌĆö an earlier `#user/mailbox/contact` form was simplified down to this single segment. **DID-preferred** (`did/contacts.ts`'s `contactIdentityKey`): an address is a replaceable relay detail, not the identity, so the permalink shouldn't be the one place that still treats it as the "real" name. Falls back to the literal address when no DID is known (most real contacts never publish one ŌĆö Gmail, Mastodon, etc.).

Because a conversation hash is now a single, shapeless segment ŌĆö the exact same shape a menu-page hash (`#account`) has ŌĆö `menuHashFromHash` disambiguates via an **explicit allowlist** (`MENU_PAGE_NAMES`) rather than "no slash = menu page" (which stopped working the moment contact hashes lost their own slash). Decoding (`main.ts`'s `matchInboxForHash`) compares via `contactIdentityKey` rather than plain string equality ŌĆö `InboxSummary.contact` drifts to whichever address most recently had traffic (see [Storage model](#storage-model)'s inbox-grouping note), so a permalink built from an older address must still match the same DID-grouped row, regardless of which of the user's own accounts it's under (a same-contact-under-two-different-logged-in-identities collision is a real but rare edge case ŌĆö first match wins). If the contact segment is a DID this browser has never resolved yet (a fresh device, or a shared link opened cold), one live DHT resolve (`resolveDidDirect`) is attempted before giving up ŌĆö the same self-healing property compose's DID input has (below), applied to permalinks.

The left-pane inbox list label (`left-pane.ts`'s `makeLpItem`) follows the same principle ŌĆö it used to prefix the row with `mailbox / contact`; now it's just the contact's name (if known) or address/group name, self info dropped entirely.

## Hamburger menu

One shared `#lp-hamburger-menu` element (`position: fixed`), triggered by two buttons:
- `#lp-hamburger` ŌĆö top-right of the right column header (desktop)
- `#lp-hamburger-left` ŌĆö top-right of the left pane header (mobile `show-left` mode)

`main.ts` attaches mouseenter/mouseleave/click handlers to both buttons via a loop. The menu is positioned via `getBoundingClientRect()` of the triggering button.

## Recurring gotcha: multi-relay identity, one session

One `email` identity can be backed by two `sessions[]` entries (mail relay + AP relay, see below). Any operation that mutates server state for "this identity" ŌĆö mark-read, delete, display-name save ŌĆö must **group affected emails by `_relay` and dispatch each group through `sessionForRelay(user, relay)`**, not just grab `sessions.find(s => s.account.email === target.user)` (which silently picks one relay's session and no-ops or fails for the other's ids). This exact bug class has recurred independently in unseen-badge marking, delete (`doDeleteInbox` in `left-pane.ts`), and display-name save ŌĆö when adding a new bulk-mutation feature, apply the relay-grouping pattern from the start.

A related trap: after a server-side mutation succeeds, the in-memory `store/` (and its IndexedDB cache) is not automatically updated ŌĆö `messages.remove(acct, id)` / `removeMessage(acct, id)` (from `vault/persist.ts`) must be called explicitly, or list views rebuilt from `store/` (e.g. `loadLeftInboxes`) keep showing stale data until a manual reload.

## Browser-local cache (`src/store/idb.ts`, `src/store/cache.ts`)

See [Storage model](#storage-model) for the full map of where data lives (this is one tier of four on the client). Always-on IndexedDB persistence for the in-memory `store/` (messages, threads, mailboxes, identities, contacts, per-relay querystate cursors), distinct from the opt-in filesystem vault (`vault/persist.ts`, requires the user to mount a directory). Loaded at boot via `loadFromCache()` (`main.ts`'s `initInner`, timeout-guarded at 3s so a hung IDB open never blocks app start) so a plain refresh has last-sync data immediately and `sync/session.ts`'s querystate-driven delta sync runs instead of a full historical re-fetch (+ re-decrypt of every PGP message). `db()` deliberately does not cache a rejected `openDB()` promise, and `deleteDB()` closes the cached connection before calling `indexedDB.deleteDatabase()` ŌĆö otherwise a lingering open connection blocks the delete (`onblocked`, silently treated as success) and queues every subsequent `openDB()` call indefinitely, which looks like "logout ŌåÆ relogin ŌåÆ nothing syncs."

## Reactions (RFC 9078)

`src/mail/reactions.ts` implements the generic IETF "Simple Mail Reactions" spec, deliberately kept out of `deltachat/protocol.ts` since the wire format isn't DeltaChat-specific. Reactions are detected both in cleartext (`isReactionEmail`, header check before PGP decrypt) and inside decrypted MIME (`isReactionDisposition`), cached on the target `Email` object (`cacheReaction`/`readCachedReaction`), and collected per-thread (`collectReactions`) into a target-messageId ŌåÆ sender ŌåÆ emoji map (latest-per-sender wins; empty body = retraction). Reaction emails are filtered out of normal message lists (`isReaction()` guards in `app.ts`'s inbox-loading functions) and rendered as chips under the target message in `thread.ts`.

## Attachments (DeltaChat-compatible)

Sent as `multipart/mixed` MIME (`pgp/crypto.ts`'s `buildMultipartBody`), matching DeltaChat's own attachment convention rather than inventing a biset-specific format ŌĆö this was a deliberate compatibility choice, not an accident. Text-only sends keep the original flat (non-multipart) body untouched. On receive, `processing.ts` converts decrypted MIME attachment bytes into `MsgAttachment[]` (`{ filename, contentType, dataUrl }`) via the shared `bytesToDataUrl` in `utils.ts` (also used by `deltachat/avatar.ts` for group/contact avatars ŌĆö one bytesŌåÆdata-URL encoder, kept in sync). Rendered in `thread.ts`: images as clickable thumbnails (opens full-size in a new tab), other files as a filename + download chip (`<a download>` against the `data:` URL).

## `scrollToFocused` (`ui/thread.ts`) ŌĆö the two-branch scroll target

Two rules, in order: (1) scroll to the bottom of the latest message; (2) **unless** the latest message alone is taller than the available viewport, in which case pin its *top* just below the sticky title row instead (its tail necessarily stays off-screen either way ŌĆö chasing it just hides the header). Rule 2 needs "is this message's own height greater than `safeHeight` (`outer.clientHeight` minus the reply-dock's covered height)" ŌĆö `last.offsetHeight > safeHeight`. A previous version instead compared the message's cumulative scroll *offset* (`lastTopInOuter`) against one viewport-height ŌĆö which is true for any long *thread* regardless of the last message's own height, and false for a single very tall message near the top of a short thread (the exact case rule 2 exists for), so it silently never fired and rule 1 always won. This bug recurred multiple times before the wrong variable was identified ŌĆö if "scroll lands at the tail of a long message instead of pinning its header" resurfaces, check this condition first before re-deriving the fix from scratch.

## Reply dock

**Visible exactly when it has a reply box in it ŌĆö never by anyone setting `display` (2026-07-14).** A day of intermittent "reply box missing / flickers and vanishes" reports came down to `renderMenuInboxImpl`, `switchInbox`, and `hideCmdPage` each writing `#reply-dock`'s inline `style.display` by hand: three call sites that had to agree on ordering, and every bug was that agreement breaking. But `style.css` already had `#reply-dock:empty { display: none }`, and `render()` (thread.ts) already either fills the dock with a `.reply-box` or clears its `innerHTML` ŌĆö **content was already the single source of truth, and `style.display` was a hand-maintained second copy of the same fact**. All three writes are gone; `renderMenuInboxImpl` clears `innerHTML` instead (nothing else would, since a menu page never calls `render()`), and the other two don't touch the dock at all. Don't reintroduce a `display` write here.

**Accepted regression: "stow the dock while scrolling up to read history" was removed, not fixed (2026-07-14, user: "ŃüØŃéīŃü»µŖĆĶĪōńÜäŃü¬ķĆĆĶĪīŃĆéń¦üŃü¤ŃüĪŃü«µĢŚÕīŚ").** It was a wanted feature, cut because keeping it needed a `markProgrammaticScroll`/`isProgrammaticScroll` timing window (`utils.ts`, since deleted) at four separate call sites plus a stale-prone `lastScrollTop` baseline ŌĆö all to avoid mistaking our own scrolls for the user's ŌĆö and kept finding new gaps anyway (native scroll-anchoring on thread-open, slow devices missing the window). `main.ts`'s scroll handler carries the full post-mortem. If re-implemented, don't key it off wall-clock windows around "was this our own scroll"; use something that can't go stale by construction, e.g. a state machine tied to actual DOM/animation completion events.

## Mobile UX patterns

- **`navFocusEnabled()`** (`left-pane.ts`, gated on `window.innerWidth > 520`) ŌĆö the keyboard-nav-cursor `.focused` class is desktop-only. It's gated at the two functions that ever add it (`syncNavFocus`, `lpFocusEl`), not at individual call sites ŌĆö earlier per-call-site guards were unreliable because several render paths (`renderThreadAccordion`, `renderLeftInboxes`, `applyLpSearch`) call `syncNavFocus()` independently.
- **Swipe gestures** (left-pane swipe-to-reveal in `main.ts`, swipe-to-delete on `.lp-item` in `left-pane.ts`) ŌĆö both use touchmove-driven direction locking: ignore movement under an 8px threshold, then lock to `'x'` or `'y'` based on `abs(dx) > abs(dy) * 1.5`, and only `preventDefault()` once locked horizontal. This is what stops a near-horizontal swipe from also fighting vertical scroll. Delete additionally live-follows the drag distance (`translateX`, capped at `SWIPE_MAX`) rather than a threshold-only reveal.
- **Mobile pane switch** (`#app.lp-enabled.show-left`) uses `transform: translateX(...)` sliding with both panes always in the DOM, not `display:none/flex` toggling ŌĆö cheaper to animate, but means elements are never inert, so anything that assumed "hidden pane = no-op" (e.g. `#lp-search` `.focus()` calls) must be explicitly gated to desktop instead.
- **Glass/blur header effect** (`#left-pane-header`, `#thread-title-row`) needs three things together or it silently degrades: (1) the header must `position: sticky` inside the actual scroll container (content has to scroll *behind* it), (2) `backdrop-filter: blur(...)` present, (3) the gradient's "opaque" stop must be a *translucent-but-strong* color (`--glass-strong-bg`, 0.88 alpha) ŌĆö a fully opaque stop hides the blur entirely since there's nothing left to show through.
- **iOS status bar / Dynamic Island**: not CSS-controllable. Governed by `<meta name="apple-mobile-web-app-status-bar-style">` in `index.html` ŌĆö `black-translucent` lets page content extend behind it; `default` makes iOS draw its own opaque native bar regardless of page CSS. Only takes effect in standalone (home-screen-installed) mode, not a regular Safari tab.
- **`env(safe-area-inset-top)` is the #1 "works in a Safari tab, breaks in the standalone PWA" trap.** In a normal Safari tab it resolves to **0** (browser chrome owns the notch); only in a home-screen PWA (with `black-translucent`, above) does it become ~59px. So any layout bug that only reproduces in the installed PWA ŌĆö content colliding with the status-bar clock, a mysterious ~59px gap, scroll position "off" ŌĆö is almost always this inset landing somewhere wrong. You cannot debug it in a browser; it's invisible there.
- **The notch inset must NOT go on `#thread-title-row`.** That row sits *between* `#past-threads` and the active thread in the DOM, so at most scroll positions it's mid-content, not at the top. Putting `padding-top: calc(... + env(safe-area-inset-top))` there made the inset render as a dead ~59px gap between past-threads and the focused card (and left the first past-row jammed under the island). For the same reason **the title row can't be frosted/translucent** ŌĆö a `backdrop-filter`/semi-transparent title reads as a floating glass band whenever it's mid-content. It must stay a flush opaque-ish surface.
- **Detect standalone with `navigator.standalone`, NOT `@media (display-mode: standalone)`.** biset has no web manifest and launches via the legacy `apple-mobile-web-app-capable` meta, so iOS does **not** match the `display-mode: standalone` media query ŌĆö it only exposes the non-standard `navigator.standalone === true`. An inline `<script>` in `index.html`'s `<head>` (runs before first paint, no flash) sets `document.documentElement.classList.add('pwa-standalone')` from `navigator.standalone || matchMedia('(display-mode: standalone)').matches`. All notch CSS is scoped to `html.pwa-standalone ŌĆ”`. Keying off this class (not the `env()` value) also avoids a browser landscape notch falsely triggering it, so the browser version stays byte-for-byte the working layout.
- **Correct notch handling (the shape that finally worked), all under `html.pwa-standalone`:** (1) `#outer` stays full-height and flush ŌĆö never pad the *scroll container* (an inset there scrolls away with content and makes `position:sticky` anchor browser-dependently). (2) `#past-threads` (always `#outer`'s first child) gets `padding-top: env(safe-area-inset-top)` to **reserve** the island height at the very top of the scrollable content ŌĆö this is what puts the first row below the island (tappable) and stops the initial scroll from tucking content under the notch. (3) `#thread-title-row` sticks at `top: env(safe-area-inset-top)`, landing directly below (4) the island strip `#right-col::before` (`position:absolute; top:0; height:env(safe-area-inset-top)`). Give the strip and the title the **same material** (`var(--bg)` @ `opacity:0.9`, no blur) so they read as one continuous surface ŌĆö mismatched textures look broken, and the title **can't** be frosted/translucent because it's mid-content at most scroll positions (reads as a floating glass band). Scope the strip to `#right-col` so the left pane (own `#left-pane-header`) is untouched.
- **`scrollToFocused` / notch interaction (two spots, both easy to miss):** (a) it derives `pastH` from `#past-threads.offsetHeight` ŌĆö subtract that element's `padding-top` (the reserved island height), or the menu/thread over-scrolls by the inset and tucks content under the title. (b) It compensates for the sticky header covering the top of `#outer`: use `titleH + stickyTop` where `stickyTop = parseFloat(getComputedStyle(titleRow).top)` (the sticky `top:` = the inset in the PWA, 0 in the browser), not `titleH` alone ŌĆö otherwise the focused message tucks under the header by the inset. Both read the live values, so both are automatically no-ops in the browser.
- **The "bottom blank strip" in the installed PWA is `100dvh` collapsing, NOT a static CSS bug ŌĆö don't chase it with padding/height math.** Symptom: no gap on a fresh launch, but after the keyboard opens once and closes, a ~59px blank strip appears at the bottom and stays until reload. Measured on device across one keyboard cycle: `window.innerHeight` (and `#app`, at `height:100dvh`) goes **852 ŌåÆ 793 and never returns** ŌĆö iOS resolves `dvh` to the *large* viewport (full screen, 852) at first paint, then permanently switches it to the *small* one (852 minus the ~59px Dynamic Island strip, 793) the first time the soft keyboard appears. `#app` follows `dvh` down and the 59px it vacates at the bottom is the blank. This wasted a long debugging arc disguised as a reply-dock/safe-area problem (every static fix ŌĆö growing `#app` by the safe-area inset, transform-trap theories, `body::after` fillers ŌĆö missed because the real variable is `dvh` changing value over time, invisible in a single snapshot). Fix: `html.pwa-standalone #app { height: 100vh }` ŌĆö in the installed PWA there's no URL bar, so the large/stable viewport is the correct full height and `vh` (== `lvh` on iOS) does not collapse. The browser tab keeps the base `100dvh` (it genuinely needs the dynamic value for the collapsing URL bar). If a bottom gap ever resurfaces here, re-measure `innerHeight` before *and after* a keyboard cycle before touching anything else.

# Storage model

biset's data lives in four places on the client and one on each relay. None of them is "the database" in the traditional sense ŌĆö each exists for a different durability/availability tradeoff, and the same JSON shape (JMAP objects, plus a couple of biset-specific ones) flows through all of them.

## Client-side

| Layer | Location | Scope | Survives |
|---|---|---|---|
| **In-memory stores** | `src/store/messages.ts`, `threads.ts`, `mailboxes.ts`, `identities.ts`, `contacts.ts` | This page load | Nothing ŌĆö plain arrays/Maps the UI reads/writes directly |
| **IndexedDB cache** | `src/store/idb.ts`/`cache.ts`, DB `biset-cache` | This browser | Page refresh/reload. Always on, no user action ŌĆö loaded at boot (`loadFromCache()`) so a plain refresh has last-sync data immediately and delta sync (querystate) runs instead of a full historical re-fetch |
| **Vault** | `src/vault/persist.ts`, File System Access API | A user-chosen folder | The browser dying entirely ŌĆö it's a real folder on disk. **Opt-in, Chromium-only** (FSAA + `FileSystemObserver` aren't implemented elsewhere) |
| **localStorage** | plain keys, this origin | This browser | Page refresh, not a cache-clear. Holds DID-specific bindings that don't fit the JMAP-object stores above (below) |

The IndexedDB cache and the vault store the **same objects** (JMAP `Email`/`Mailbox`/`Identity` plus biset's `Card`), just at different durability tiers ŌĆö every `flush*` in `vault/persist.ts` writes to the IndexedDB cache unconditionally and to the vault file only `if (vaultHandle)`. Vault files: `.data/messages/<account>/<id>.json`, `.data/threads/<threadId>.json`, `.data/mailboxes.json`, `.data/identities.json`, `.data/contacts.json`, `.data/submissions/<id>.json`, plus the rendered Markdown mirror (see README's "Vault structure"). Same shape in IndexedDB, one object store per JSON file (`STORES.messages/threads/mailboxes/identities/contacts`, `mailboxes`/`identities`/`contacts` as a single out-of-line-keyed blob matching their one-file-per-account vault counterpart).

**localStorage** (not IndexedDB ŌĆö it's small, synchronous, DID-specific keys, not a general object store):
- `biset_did_addr:<address>` ŌĆö addressŌåÆDID TOFU binding (cache only; re-derivable from a DNS TXT lookup).
- `biset_did_contact:<address>` ŌĆö `ContactCache { did, address, relays, protocol?, name?, lastChecked? }`, one entry per address this browser has resolved via `did/discovery.ts`. This is the per-address working cache `freshestAddressFor`/`protocolForContact`/`refreshContact`'s TTL gate all read from directly; `did/contacts.ts`'s `buildCardForDid` consolidates every entry sharing a DID into one `Card` for the stores above.
- `biset_did_name:*`-shaped keys don't exist ŌĆö a display name is just JMAP `Identity.name` (see below), not a separate DID-specific setting.

## Server-side (each relay, independently)

Every relay (`go-jmapsmtp`, `go-jmapap`) owns a `jmapserver.Store` per account, rooted at `dataDir/<domain>/<localpart>/`:

| File | Contents |
|---|---|
| `messages/<id>.json` | One JMAP `Email` object per message |
| `mailboxes.json` | The account's `Mailbox` list |
| `delta.json` | `Email`/`Mailbox`/`EmailSubmission` change-state counters (JMAP `/changes` support) |
| `identities.json` | JMAP `Identity` objects ŌĆö the account's own send-identities (From name, signature, ...) |
| `contacts.json` | JSContact (RFC 9553) `Card` array ŌĆö **this account's** resolved contacts, DID-rooted (see below) |
| `peers/<addr>.pgp` | (jmapsmtp only) cached Autocrypt public keys, learned from inbound mail or WKD prefetch |

`contacts.json` is the server-side half of the client's `ContactCache`/vault `Card` ŌĆö a write-through, not a second source of truth: `did/discovery.ts`'s `syncContactCard` pushes one `Card` at a time (`PUT /contacts/<uid>`) to every one of the account's own relays whenever it resolves or refreshes a contact, and pulls the full list (`GET /contacts`) once at session start (`pullOwnContacts`) so a fresh device/browser ŌĆö or any non-Chromium browser, where the vault doesn't exist at all ŌĆö inherits previously-resolved contacts instead of starting blind. Neither the vault, IndexedDB, localStorage, nor this server copy is the *ground truth* ŌĆö the contact's own signed DID document is; all four are just durability tiers for what's already been resolved from it (see [Identity layer (DID)](#identity-layer-did)).

`go-jmapserver/contacts.go` implements the shared `Card` type + `ReadContacts`/`PutContact` (plain file read/upsert-by-uid, no locking ŌĆö same risk level as `mailboxes.json`, which has none either) + `RegisterContactsEndpoints` (the `GET`/`PUT /contacts` HTTP routes, taking each relay's own `authenticate` function as a callback since jmapap/jmapsmtp resolve accounts differently).

A `Card`'s `uid` is `urn:uuid:` + a **UUIDv5** (RFC 4122 ┬¦4.3) of the DID string, under a fixed biset namespace constant ŌĆö deterministic, so "does this DID already have a Card" is a `uid` lookup, not a separate DIDŌåÆuid index. Fields map onto native JSContact properties wherever one exists: `cryptoKeys` holds the DID itself (a DID is a URI by construction, no vendor extension needed), `links` holds the contact's current relay endpoints, `emails` holds every address known for that DID, `name.full` holds their self-asserted display name if their DID document carries one. Only `biset.md:verifiedAt` (a bookkeeping timestamp) needed the vendor-extension form the spec requires for non-standard properties.

(`go-didanchor`'s own `_did/` claim registry is a *different* store, for a different question ŌĆö "is this address/DID combination globally unique" ŌĆö not covered here; see [Identity layer (DID)](#identity-layer-did).)

## The Email/Mailbox/Thread hierarchy

`messages/<id>.json` (and its client-side counterparts in [Client-side](#client-side)) hold a standard **JMAP `Email` object (RFC 8621)** ŌĆö already a *parsed* representation of the raw RFC 5322 message, not the raw bytes themselves:

```
Account
 ŌööŌöĆ Mailbox (folder; parentId forms a tree)
      "a named set of Email objects" ŌĆö totalEmails/unreadEmails etc. are server-set aggregates, not stored per-message
      Ōöé
      ŌööŌöĆ Email (mailboxIds: the set of Mailboxes it belongs to ŌĆö can be more than one)
           Ōö£ŌöĆ threadId ŌåÆ Thread (just an emailIds[] array, sorted by receivedAt ŌĆö a thread holds no data of its own)
           Ōö£ŌöĆ blobId ŌåÆ the raw RFC 5322 byte stream itself (the actual "raw data" ŌĆö fetched separately, on demand)
           Ōöé
           ŌööŌöĆ bodyStructure: EmailBodyPart (the full MIME tree, as-is ŌĆö multipart/* parts recurse via subParts)
                Ōö£ŌöĆ bodyValues: Record<partId, EmailBodyValue> ŌĆö the decoded text of each text/* part, keyed by partId back into bodyStructure
                Ōö£ŌöĆ textBody / htmlBody: EmailBodyPart[] ŌĆö a pre-picked, depth-first "what to actually render" list (pointers into bodyStructure, not a data copy)
                ŌööŌöĆ attachments: EmailBodyPart[] ŌĆö everything else non-multipart that isn't in textBody/htmlBody
```

So `Email` is already the *parsed* form ŌĆö MIME structure, decoded text, attachment list all pulled apart for the client. The literal, unparsed wire bytes are a separate thing entirely, reachable only via `blobId`; biset's whole pipeline (rendering, PGP decrypt, DeltaChat header handling) works against the parsed `Email` object, never the raw blob. `Thread` sits *above* `Email` as pure grouping ŌĆö it carries no content of its own, just an ordered list of member ids.

**biset's own non-standard fields, stamped onto the `Email` object once at ingest** (`sync/session.ts`) and persisted client-side alongside it (vault/IndexedDB ŌĆö the relay itself never sees or stores these, they're added *after* fetching over JMAP):
- `_account` ŌĆö owning account (`email\0serverUrl`), since two accounts on the same relay can produce colliding JMAP ids.
- `_relay` ŌĆö which relay (serverUrl) this Email arrived through.

That's it ŌĆö a third field, `_identity` (the owning identity's DID-or-email grouping key), used to exist here too but was removed as redundant (2026-07-14): `store/messages.ts`'s `forIdentity(identity)` now resolves which `_account`s belong to an identity **dynamically**, via `relaysForId(identity)` (`context.ts`'s live `sessions`), rather than trusting a value stamped once at ingest-time. Same organize-don't-duplicate principle as contact-DID grouping (below) ŌĆö `_account` is the only fact that needs to be *stored*; "which identity currently claims this account" is answered fresh every query, so a lazily-DID-migrated identity's older messages (ingested before the account had a DID) are picked up correctly with no re-stamping needed.

These ride alongside the standard fields in the exact same JSON file ŌĆö `messages/<id>.json` is never a biset-specific shape, just a standard `Email` with a couple of extra top-level keys a generic JMAP client would simply ignore.

# Identity layer (DID)

Every biset identity is rooted in a client-generated, rotation-less **ed25519 key**, expressed as a **`did:dht`** identifier and published as a signed record on the BitTorrent Mainline DHT (via [Pkarr](https://github.com/pubky/pkarr)/BEP44). The DID document lists the identity's current relays (`service`) and address(es) (`alsoKnownAs`) ŌĆö resolving DID ŌåÆ current state is what makes moving relays, adding a relay run by a different operator, or losing a domain not mean losing your identity. Design rationale and decision history live in `DID.md` / `PLAN.md` (kept out of the repo); this section is the current-state summary.

## Why did:dht?

The choice is **Mainline DHT** for economic reasons: Mainline has no protocol-level incentive layer (no token, no miners/validators) and biset needs none either ŌĆö the network's ~10M nodes already exist, already run 24/7, and already cost their operators nothing extra to keep routing.

**Joint production.** BitTorrent clients don't run a DHT node *in order to* support the DHT ŌĆö they run one as an inseparable byproduct of what they actually want, which is to find peers for a torrent someone is downloading. Maintaining routing-table health is the classic joint-production case (grazing cattle for meat yields hide as a byproduct): the "service" did:dht rides on was never built or staffed as a service, it's the residue of millions of self-interested downloads.

**Positive externality, free-ridden deliberately.** Those node operators have no idea they're carrying `did:dht` traffic and are paid nothing for it ŌĆö biset is a pure free rider on an externality nobody had to intend into existence. This is *why* the design can skip everything a blockchain needs to internalize consensus-safety costs (staking, fees, block rewards): there is no global order to agree on. Each record is self-authenticating (signed by the DID's own key ŌĆö verified client-side, no network-wide consensus required, see `didbind.go` above) and self-expiring (BEP44 TTL, ~2h without re-announcement ŌĆö see the delete/`Gateway.Forget` discussion below), so there's no persistent-storage externality being pushed onto anyone either. The free ride isn't just "use someone else's network," it's "use a resource whose marginal cost per record is already indistinguishable from zero."

**Consequence for biset's own architecture:** biset never had to stand up or fund DHT infrastructure. The optional `pkarr/` gateway in `go-jmapserver` (`PKARR_GATEWAY=1`) is a convenience cache-and-republisher sitting *on top of* the DHT, not a replacement for it or something biset is on the hook to keep alive ŌĆö if every biset-run gateway disappeared tomorrow, any still-referenced DID would keep resolving through the DHT directly (or any other unrelated Pkarr-aware relay) until its own ~2h TTL lapsed. The tradeoff is the flip side of free-riding on an externality: there's no SLA, and no vendor to escalate to if resolution degrades ŌĆö a real risk for hosted convenience layers, but a much smaller one for Mainline DHT itself, precisely because its survival was never coupled to any single company's funding or roadmap.

## Where it lives

**Client (`src/did/`):** `seed.ts`/`keys.ts` (BIP39 mnemonic ŌåÆ SLIP-0010 root key + sub-keys ŌĆö Nostr via NIP-06, DIDComm's `_k1` at `m/1'`, PGP reserved at `m/2'`), `document.ts`/`dns.ts`/`packet.ts` (DID document Ōćä DNS-RR Ōćä signed BEP44 payload), `chain.ts` (multi-record documents, below), `resolver.ts` (DHT resolve/publish over gateway HTTP), `discovery.ts` (contact addressŌåÆDID resolution, bidirectional-verified), `sync.ts` (**self**-resolution ŌĆö the account holder's own reconnection follows their DID's current service list, not a static cached URL), `provision.ts`/`binding.ts` (signature-based DIDŌåÆrelay binding, no secrets sent to third-party relays), `restore.ts` (full identity recovery from the 24-word phrase alone), `didcomm/` + `peer.ts` ([DIDComm messaging](#didcomm-messaging) below).

`store.ts` persists only *derived* keys (root/Nostr/`_k1`) and the mediator a `_k1` is registered with ŌĆö **never the master seed**, which is used transiently at login and discarded. New key material follows that rule: derive it, store the result, let the 24 words be the only backup.

**Relay side, shared (`go-jmapserver`):** `didbind.go` (`VerifyDIDBinding` ŌĆö pure ed25519 signature check, no network dependency), `anchorclient.go` (`AnchorClaim` ŌĆö HTTP client to the anchor service), `cloudflare.go` (`CloudflareAnchor` ŌĆö DNS TXT record read/write), `didindex.go` (per-relay local index: which addresses on *this* relay belong to which DID ŌĆö organizational only, does not merge storage; see below), `pkarr/` (the DHT gateway, `GET/PUT /pkarr/<key>`, opt-in per relay via `PKARR_GATEWAY=1`).

**anchor (`go-didanchor`):** a separate, minimal, entirely optional binary ŌĆö the identity registry. Two jobs only: (1) claim/verify `localpart+domain ŌåÆ {fingerprint, did}` (cross-relay split-identity detection ŌĆö the same address can't be claimed by two different keys), (2) when a DID is present, keep its `_did.<localpart>.<domain>` DNS TXT record current via the Cloudflare API. Deliberately kept this narrow ŌĆö no messaging protocol, no live traffic ŌĆö so its trust footprint (it holds a DNS-write credential) stays small. jmapap and jmapsmtp are equal HTTP clients of it, pointed at via `anchor_url` in config; unset = "anchorless" (see below).

**Anchorless / with_anchor is a config-driven spectrum, not two codebases.** `/account/provision`'s `did`/`did_sig` fields are optional ŌĆö a client that omits them gets a plain JMAP account (no discovery, no portability, same as a classic mailbox). If a DID is present but no `anchor_url` is configured, binding is still cryptographically verified (pure math, no anchor needed) but there's no cross-relay coordination or DNS publication. Only with both DID *and* a configured anchor do you get the full picture (discoverable, portable, split-protected).

**AddressŌåÆDID lookup is DNS, not a bespoke endpoint.** A `_did.<localpart>.<domain>` TXT record (`did=<did>`), resolved client-side via DNS-over-HTTPS (Cloudflare, Google fallback) ŌĆö mirrors ATProto's `_atproto.<handle>`. This decouples "who answers for an address's identity" from "who runs its relay": DNS is a commodity, swappable, and self-hostable by whoever owns the domain, which a bespoke relay-hosted endpoint never was. Custom (BYO) domains (`src/ui/custom-domain.ts`, `go-jmapsmtp`'s `/domain/verify-token` + `/domain/add`) extend this to end users ŌĆö host mail on an existing relay while keeping the domain (and hence the exit) under your own control.

**Data model: DID-rooted organization, address-rooted storage.** Two addresses under one DID each get their own fully independent JMAP `Store` ŌĆö server-side storage is never merged across addresses. What *is* DID-rooted is a local index (`didindex.go`'s `data/_did_local/<did>`) recording which addresses on a given relay trace back to which DID, exposed read-only at `GET /identity/local/<did>` (public/unauthenticated ŌĆö an address is meant to be discoverable given its DID, same sensitivity as anything else in this layer). The unified inbox view a user actually sees comes entirely from the *client's* existing identity-by-DID grouping (`identityKey() = did || email` in `context.ts`) ŌĆö a deliberate choice after an earlier attempt at physical storage-sharing introduced a real duplicate-message race with no corresponding benefit the client-side grouping didn't already provide. See DID.md's "Data-model inversion" section for the full story.

**Contact-DID grouping applies the same lesson to correspondents, not just the user's own accounts (2026-07-14).** `did/contacts.ts`'s `contactIdentityKey(address)` groups a *contact's* addresses by DID exactly the way `identityKey()` already grouped the user's own ŌĆö organizationally, via the `Card`s in [Storage model](#storage-model), never by merging message data. `app.ts`'s `loadInboxSummaries` keys inbox rows on it (so a contact who migrated relays mid-conversation stays one row, not two) and `getInboxEmails` widens its address filter via `allKnownAddressesFor` to match. `refreshContact`'s DHT re-check is TTL-gated (currently 1h; was 24h) ŌĆö first contact always resolves, but re-checking on literally every send would be wasted DHT/gateway load for a fact (a migration) that changes rarely; the TTL only throttles the *check*, not how often it's triggered (opening/sending in a conversation), so a contact not spoken to in weeks still catches up fully the next time they are.

**Protocol-aware send routing (`relayForSend` in `ui/shell.ts`).** `freshestAddressFor` only swaps the *address* a reply goes to, not which of the user's own relays sends it ŌĆö so a contact who drops ActivityPub entirely for mail-only (or vice versa) would otherwise still get routed through the old protocol's relay, which can't reach a differently-shaped address at all. `ContactCache` and `Card` now also carry `protocol` (from the DID document's matching `service.protocol`); `relayForSend` compares the target's current protocol against the conversation's established relay and, on a mismatch, switches to whichever of the user's own relays (`relaysFor`, same DID) speaks the right one ŌĆö falling back to the established relay if the user has no matching relay of their own.

**Self-asserted display name (`DidDocument.name`, `document.ts`).** A did:dht additional-property extension, same pattern as `service.protocol`/`.address` ŌĆö base64url-encoded in the root DNS record (so an arbitrary name can never collide with the `;`/`,` field separators the rest of the document's encoding relies on). Reuses the *existing* "Change display name" JMAP `Identity.name` (no separate DID-specific name to manage) ŌĆö saving it there also triggers a republish. Purely a UX label, same trust level as any social profile's display name, never verified. Surfaces via `did/contacts.ts`'s `displayLabelFor` ŌĆö a 3-tier fallback (self-asserted name ŌåÆ `did~xxxx`, the last 4 chars, if a DID is known but has no name ŌåÆ the literal address) used identically by both the left-pane inbox list (`makeLpItem`) and the conversation header (`thread.ts`'s `conv-to`) ŌĆö self info (which mailbox/relay it's under) is never part of either label, only the contact's own identity. The small `[DID]` pill next to `conv-to` stays a plain "DID-mediated" marker, since the main text already carries the name/short-DID.

**Composing directly to a DID (`resolveDidDirect`, `discovery.ts`).** Complements `refreshContact` (which starts from a known address) for contacts shared via QR code / profile link / permalink, where no current address is known at all ŌĆö resolves the DID's document, applies the same reverse-binding verification (a claimed address is only trusted once *that* address's own anchor points back to the same DID), and caches the result exactly like an address-first resolve. Wired into both the compose-new recipient field (`left-pane.ts`'s `attachPrefetch`, with a spinner in the same slot the AP/mail protocol pill occupies ŌĆö no timeout, a spinner that never stops *is* the failure signal) and, redundantly, the Send button itself (`resolveDidRows`) ŌĆö resolution is async and nothing stops a user clicking Send before the blur handler finishes, so the button re-resolves in place rather than risk sending to a raw, unresolved `did:...` string.

**One client session = one identity (2026-07-14).** biset can *store* credentials for several distinct identities (DIDs) at once ŌĆö `loadStoredAccounts()` is a flat, unscoped list ŌĆö but only ever loads **one** of them into `sessions[]`. `context.ts`'s `accountsForActiveIdentity` narrows the stored list down to whichever identity is currently active (a persisted `did || email` marker, `getActiveIdentity`/`setActiveIdentity`) before `main.ts`'s boot sequence calls `initSession` on anything; if no active identity is set yet, or the one that was active no longer has any stored accounts (logged out of it entirely), it adopts whichever identity IS present. Because `app.ts`'s `loadInboxSummaries` and everything downstream already iterate `identityIds()` (the *distinct* DIDs found in `sessions[]`) rather than assuming exactly one, this required no changes to the merging logic itself ŌĆö constraining the input to one identity was enough.

Switching identity is currently **logout, then log back in as the other one** (the "Log out" account-menu action already removes a whole identity's stored accounts; the next login naturally becomes the new active identity via the fallback above) ŌĆö deliberately not building a Gmail-style account switcher yet. Two related things are explicitly deferred, not forgotten:
- **A real "switch identity" UI** (pick from previously-used identities without re-entering a password, the way Gmail's account switcher does) instead of full logout/login.
- **Distinguishing "add a relay to my current identity" from "sign in to a different identity entirely"** in the account-add flows ŌĆö the boot-time filter above just silently leaves an inactive identity's stored credentials unloaded until it somehow becomes "active"; no UI yet helps a user manage multiple stored identities explicitly.

**"Log in to existing account" now refuses a genuinely different identity (2026-07-14, user-reported).** Was the one real remaining gap in the above: logging in with an email whose account belongs to a different DID silently merged its session into the active identity's `sessions[]`, violating one-session-one-identity ŌĆö reported live (logged into `y@biset.md`, DID `6oien8gcŌĆ”`, then successfully "logged in" with `y@orillo.org`, whose account is bound to an entirely different DID `5chjsir3ŌĆ”`). Root cause: `initDid(email, masterSecret)` derives a DID purely from *that account's own* unsealed seed ŌĆö correct in isolation, but nothing compared the result against whatever identity was already active before merging. Fixed in the login form's submit handler (`onShowAccounts`, `left-pane.ts`): when `sessions.length > 0`, compares the new account's `did || email` against `identityIds()[0]` (the active identity's key) *before* the self-resolution/merge steps run, and refuses with "This account belongs to a different identity ŌĆö log out first to switch" on a mismatch. A same-identity relay logged in through this form (rather than "Add a relayŌĆ”") still works fine, since its derived DID naturally matches.

**"Add a relayŌĆ”" (2026-07-14), not "Move to another relay."** Nothing is ever removed ŌĆö the old name implied migration, but the operation only ever adds a new relay/address under the *same* DID, the existing account stays connected. This is the same operation as "Add my own domain" when already logged in: both recover the active identity's existing DID + master secret via password and `provisionAccount` a new address under it, rather than minting a fresh identity. Factored the shared part out of what was duplicated inline across `left-pane.ts` and `custom-domain.ts` into `did/provision.ts`'s `unsealCurrentIdentity(identityEmail, password)` ŌĆö fetches the *existing* envelope from an already-connected session of that identity, unseals it with the password, returns `{ did, rootPrivateKey, masterSecret, kek, envelope }` ready to pass into `provisionAccount`. A `hexToBytes` helper duplicated across 5 files was consolidated into `utils.ts` as part of this cleanup (`did/contacts.ts`'s unrelated dash-stripping UUID variant kept separate).

**Per-relay card menu: "Change password" / "Delete account" / "Log out" (2026-07-14).** "Log out" used to be a hidden whole-identity wipe (removed every relay/address sharing the card's DID, however many ŌĆö surprising, since it read as a per-card action), the counterpart to per-card "Remove this relay" which only ever touched the one card. That distinction is gone: "Log out" now IS the per-relay removal (renamed, `removeRelayLocally()` in `left-pane.ts`) ŌĆö local-only, server-side account untouched, and if it happens to be the identity's last remaining relay that already amounts to a full sign-out, so no separate bulk action is needed. **"Delete account"** is new and different in kind: it actually deletes the account's data server-side (messages, mailbox, envelope ŌĆö `POST /account/delete` in `go-jmapsmtp`/`go-jmapap`, `cryptenv.ts`'s `deleteAccountOnRelay`), gated behind a native `confirm()` given it's irreversible. The endpoint mirrors `registerDidUpdate`'s pattern deliberately: no email/account identifier in the request body at all ŌĆö target account comes only from the Basic Auth credential (`authenticate(r, dataDir)`), so it can never act on anyone else's account. Cleanup mirrors the existing `purgeInactiveAccounts` maintenance job (same map deletions under `h.mu`, same `os.RemoveAll`), just on-demand for one account instead of a periodic sweep.

**Deleting a whole identity, not just one relay (2026-07-14).** The per-card "Delete account" above only ever touches one relay/address. The identity heading's hamburger menu (`openIdentityMenu`) got its own "Delete account" that loops `deleteAccountOnRelay` over every stored endpoint sharing the DID, then does the same local cleanup `removeRelayLocally` does for a single relay (stored accounts, sessions, PGP keys, notif prefs, `clearIdentityCache`) ŌĆö across all of them at once. A single account-delete call's optional `{"did":...}` body field now does three cleanup jobs, not one:
1. **`jmapserver.RemoveLocalDID`** ŌĆö drops the address from this relay's own `_did_local/<did>` index (unchanged from the per-relay case above).
2. **`jmapserver.AnchorRelease`** (new) ŌĆö tells the identity anchor (`go-didanchor`) to forget its claim on this `(domain, localpart)` (`releaseIdentity` there, the delete counterpart to `claimIdentity`). Without this, the anchor would keep the claim forever and reject any *future* registration of the same address ŌĆö by anyone, including the original owner under a fresh identity ŌĆö as a false split-identity conflict. Verified live: provision identity A ŌåÆ delete ŌåÆ anchor claim gone (404) ŌåÆ a genuinely different identity B can immediately claim the same address. Does **not** remove the `_did.<localpart>.<domain>` DNS TXT record `CloudflareAnchor` may have written (no delete method exists there yet) ŌĆö a known, lower-severity gap, since the DID document that record points at fades from the DHT on its own (next point).
3. **`pkarr.Gateway.Forget(pubkey)`** (new) ŌĆö if this relay runs a pkarr gateway (`PKARR_GATEWAY=1`), evicts the DID's record from its in-memory republish cache. This turned out to matter more than expected ŌĆö see below.

**How long does a deleted identity's did:dht record actually take to disappear from the DHT?** Investigated directly rather than assumed, since the answer matters for anyone relying on "delete = gone": **BEP44's own spec** ("Expiration" section) says an item MAY expire in ~2 hours without re-announcement, SHOULD be re-announced hourly to stay alive ŌĆö Pkarr's own docs describe the same ("records are ephemeral, the DHT drops them after hours"). *But* biset's own pkarr gateway (`go-jmapserver/pkarr/gateway.go`) caches every record it has ever `Get` or `Put` in memory (`g.cache`, keyed by pubkey) and its `republishLoop` re-puts everything in that cache every 30 minutes, **forever ŌĆö nothing ever evicted an entry before this change**. So the real, previously-undocumented answer was: a deleted identity's record would keep getting actively re-published by biset's own gateway indefinitely (until that relay process happened to restart), not fade in ~2 hours at all ŌĆö the "hours" figure only ever applied to *other, independent* DHT nodes that aren't actively being told to keep it alive. `Gateway.Forget` (new) closes this: once called from account-delete, this gateway stops being that indefinite re-announcer, and the ~2-hour BEP44 figure becomes accurate again. Cuts both ways as a design note: the same unbounded-cache behavior is *why* a live, still-used identity stays resolvable indefinitely across any single relay's downtime shorter than a restart ŌĆö losing every one of an identity's relays simultaneously for longer than that is what actually risks the ~2-hour DHT-wide decay DID.md already describes ("neglect = temporary unresolvability, never death" ŌĆö true only as long as at least one relay, or its gateway's cache, is still republishing).

**A document too big for one BEP44 record spills into a chain of them (`did/chain.ts`, 2026-07-16).** A BEP44 value caps at 1000 bytes, which fits ~4 relays once `_k1` + a DIDComm service are present. Past that the signer threw, `publishOne` swallowed it, and the identity **silently stopped republishing** ŌĆö decaying out of the DHT ~2h later and becoming unresolvable with no signal anywhere. Now `splitIntoChain` spills the overflow into further did:dht records, each named by the previous one's `ext=` field; `resolve`/`publishDocument` handle it internally, so callers still see one logical document. Chosen over BEP44 `salt` (Pkarr's relay protocol ŌĆö the only path a browser has to the DHT ŌĆö has no salt field, so it would mean extending two specs and giving up public gateways) and over an external store (which trades the DHT commons for a service someone must fund). Continuation keys come from the **root private key** via HKDF, not the seed ŌĆö `publishDocument` never holds the seed, and the 24 words still restore the whole chain. Documents that already fit publish byte-identically; resolvers that ignore `ext=` still get a valid document with the first N services. Verified live: 20 relays across 3 records, resolved back in order.

**The cap is on the *bencoded* form, so the real limit is 996 bytes, not 1000 (`packet.ts`).** BEP44 says nodes may reject a put whose *bencoded* `v` exceeds 1000 bytes, and bencoding prefixes `<len>:` ŌĆö so a 999-byte packet is 1003 and gets dropped. Nodes don't answer with an error; they simply never respond (a Pkarr relay surfaces this as `Publish query timed out with no responses`). A plain `>= 1000` check therefore passed 997ŌĆō999B documents straight into silent discard. Bisected against a live gateway: 996B (=1000 bencoded) accepted, 998B not.

## DIDComm messaging

*Status (2026-07-16): plumbing verified end-to-end, not yet in the inbox UI.*

**DIDComm v2 over the same `did:dht` identity** ŌĆö a lighter bisetŌåöbiset channel than routing a DM through ActivityPub. Working end-to-end against the real DHT (send, mediator-routed receive, and mediator registration), but deliberately **not wired into the normal inbox/thread UI yet**: it lives on a `/didcomm` debug page (`ui/didcomm-debug.ts`).

| Path | Role |
|---|---|
| `src/did/didcomm/crypto.ts` | JWE: authcrypt (`ECDH-1PU+A256KW`) / anoncrypt (`ECDH-ES+A256KW`), both `A256CBC-HS512` |
| `src/did/didcomm/coordinate.ts` | Mediator Coordination 2.0 ŌĆö `mediate-request`, `keylist-update` |
| `src/did/didcomm/send.ts` | authcrypt to the recipient, anoncrypt-wrap in a Forward for their mediator, POST |
| `src/did/didcomm/pickup.ts` | Pickup 3.0 ŌĆö `status-request` / `delivery-request` |
| `src/did/didcomm/register.ts` | Publish `_k1` ŌåÆ register with a mediator ŌåÆ republish with the DIDCommMessaging service |
| `src/did/didcomm/resolve.ts` | Resolve a recipient DID (either method) to one document shape |
| `src/did/peer.ts` | did:peer:2 encode/decode ŌĆö interop fallback only, see below |

**Crypto is hand-rolled in TypeScript, and that is not incidental.** The obvious move is sicpa-dlab's `didcomm`/`didcomm-node` (the Rust reference implementation via wasm-bindgen), and it was rejected: both packages fetch a separate ~0.9MB `.wasm` at runtime, which **breaks the single-HTML-file/`file://` architecture** this whole app is built around ([Build flow](#build-flow)). So `crypto.ts` builds the JWEs directly on `@noble/curves` + `@noble/ciphers`. The constructions (ConcatKDF byte layout, ECDH-1PU's `ZeŌĆ¢Zs` and its cc_tag-in-pub_info step) were read out of `hyperledger/aries-askar`'s source ŌĆö the crate didcomm-rust itself uses ŌĆö not reconstructed from memory, and checked against the RFC/draft test vectors plus a live round-trip with the real `didcomm-node` library in both directions. `enc` is always `A256CBC-HS512`; unpack rejects anything else rather than half-supporting it. **Don't add `didcomm`/`didcomm-node` back as dependencies** ŌĆö they were removed once they had no importer.

**Transport identity: `did:dht` + `_k1`, with did:peer only as an interop fallback.** `_k1` is a seed-derived X25519 key at its own SLIP-0010 path (`keys.ts`'s `deriveDidCommKey`, `m/1'`) ŌĆö deliberately *not* a birational conversion of the root ed25519 key, since one secret must not both sign Pkarr puts and do key agreement. It's published as the document's `_k1` verification method alongside a `DIDCommMessaging` service. did:peer exists (`peer.ts`) only for mediators that can't resolve `did:dht` ŌĆö every third-party mediator surveyed. It is **not** a second identity: biset users have one `did:dht` and nothing else.

**A DIDComm-capable document needs no extra exchange step.** address ŌåÆ (DNS anchor) ŌåÆ `did:dht` ŌåÆ (DHT) ŌåÆ `_k1` + service is already the discovery path, so contact over mail/AP is enough to make DIDComm reachable. Carrying DIDs in mail headers (Autocrypt-style) was considered and rejected: a DNS anchor is trustworthy because a *third party* (the domain) attests it, whereas a header would make both directions the sender's own claim and collapse `verifyBinding`'s bidirectional check into a circle.

**Mediator:** `~/didmediator` (separate repo, TypeScript/Bun) ŌĆö Coordination + Routing + Pickup, resolving both did:dht and did:peer. Its own ARC.md carries the server-side gotchas.

# DeltaChat / chatmail interoperability

biset is a JMAP mail client that interoperates with DeltaChat / chatmail via group chat, E2EE, and securejoin. DeltaChat-specific protocol knowledge is isolated in one place.

## Layer structure

- **`src/deltachat/protocol.ts`** ŌĆö all `Chat-*` header knowledge, Autocrypt/Gossip, protected header generation, grpid generation. No `Chat-*` magic strings anywhere else.
- **`src/deltachat/securejoin.ts`** ŌĆö securejoin v3 (setup-contact), inviter (Alice) side.
- **`src/pgp/crypto.ts`** ŌĆö E2EE / symmetric crypto primitives. Symmetric message construction for securejoin lives here.
- **`src/sync/session.ts`** ŌĆö on receive: decrypts PGP, promotes protected headers, learns Autocrypt keys, drives securejoin handshake (`maybeHandleSecurejoin`).

## Message model

- **DeltaChat hides all headers inside the ciphertext** (recipients as `To: hidden-recipients:;`, Autocrypt inside). The receiver must decrypt before it can read Chat-Group-ID or the sender's Autocrypt key. `session.ts` decrypts and promotes inner headers.
- **biset sending**: the plaintext body contains armored PGP. `go-jmapsmtp` wraps it as PGP/MIME (RFC 3156), injects Autocrypt + Chat-Version headers on the outside, and DKIM-signs. The outer Autocrypt enables opportunistic E2EE even before securejoin completes.
- **Protected headers**: `Chat-*` / `Secure-Join*` / `Autocrypt(-Gossip)` go inside the ciphertext. DeltaChat ignores cleartext copies and trusts only the inner values (RFC 9788 family). `buildProtectedHeaders` handles this.

## securejoin v3 (biset = inviter Alice)

Invite URL (no QR needed, `newInviteUrl`):
```
https://i.delta.chat/#<FP uppercase 40hex>&v=3&i=<invitenumber>&s=<auth>&a=<addr>&n=<name>
```
invitenumber/auth are `create_id()`-equivalent (18 bytes URL-safe base64), persisted in localStorage.

Handshake (driven by `maybeHandleSecurejoin` on receive sync):
1. BobŌåÆbiset **`vc-request-pubkey`** (symmetric / no key). password=`securejoin/<biset FP>/<auth>`. biset tries symmetric decrypt against all active invite auth values.
2. bisetŌåÆBob **`vc-pubkey`** (symmetric + signed, inner Autocrypt = biset key). Bob learns biset's key.
3. BobŌåÆbiset **`vc-request-with-auth`** (normal E2E/PKESK, Bob key in Autocrypt, Secure-Join-Auth/Fingerprint). biset learns Bob's key and verifies auth/FP.
4. bisetŌåÆBob **`vc-contact-confirm`** (E2E, Bob key in gossip to pass Bob's self-gossip guard). Bob fires `SecurejoinJoinerProgress::Succeeded` ŌåÆ UI advances from "Establishing connection" to compose.

Processed securejoin messages are not stored ŌĆö they are destroyed via JMAP.

## Three non-obvious interop traps (confirmed at byte level)

1. **Read-only group**: if the grpid doesn't satisfy DeltaChat's `validate_id` (URL-safe base64, 11ŌĆō32 chars), Chat-Group-ID is ignored and the chat becomes a read-only ad-hoc group. `crypto.randomUUID()` (36 chars) fails this. ŌåÆ `newGroupId()` produces 18-byte base64url (24 chars).

2. **Missing CRC24 ŌåÆ `523 Encryption Needed`**: chatmail's filtermail expects an `=CRC24` suffix on armored PGP and strips everything from the last `=` onward. openpgp.js **omits CRC24 for AEAD (SEIPDv2)**, so without a checksum the base64 padding `=` gets stripped, causing decode failure ŌåÆ "not encrypted" rejection. ŌåÆ `ensureArmorChecksum` appends RFC 9580 ┬¦6.1 CRC-24 (init `0xB704CE` / poly `0x1864CFB`).

3. **S2K iterated ŌåÆ rPGP decrypt rejection**: DeltaChat's `check_symmetric_encryption` (core `decrypt.rs`) only accepts **`Salted` S2K** in symmetric SKESKs. openpgp.js only produces `Iterated` for password encryption and ignores `config.s2kType: salted`. ŌåÆ `symmEncryptSignMime` builds a **manual Salted-S2K SKESK**: derives `K = SHA256(saltŌĆ¢password)[:16]` as session key, lets openpgp.js produce SEIPDv1, then strips the Iterated SKESK and prepends a hand-crafted v4 SKESK (AES-128, Salted type 1, SHA-256, no ESK).

# ActivityPub federation & E2EE roadmap

biset federates over ActivityPub through the **`go-jmapap`** relay (a JMAPŌåöAP bridge, sibling of `go-jmapsmtp`). Relays are independent and share libraries, not state ŌĆö there is no core server. One identity can be served by both the mail relay and the AP relay (and more, after a move ŌĆö see [Identity layer (DID)](#identity-layer-did)); biset merges them into one inbox client-side (`accountKey` = `email\0serverUrl` to avoid JMAP id collisions across relays, `messages.forIdentity()` resolving `did || email` ŌåÆ its current accounts dynamically ŌĆö see [The Email/Mailbox/Thread hierarchy](#the-emailmailboxthread-hierarchy)).

## Identity anchor (split-identity prevention)

Because the relays are independent, each owns its `<localpart>` namespace on its own disk ŌĆö so the same address could be provisioned by two different people on two relays (different keys), a **split identity** exploitable for fediverse impersonation. This is now handled by the standalone **anchor** service (`go-didanchor`) rather than by `jmapap` itself ŌĆö see [Identity layer (DID)](#identity-layer-did) above for the current design (DNS-based addressŌåÆDID lookup, the anchor's narrow claim-registry role, and the anchorless/with_anchor spectrum). The anchor fingerprint/key is still the natural root for MLS credentials later, per the identity/MLS discussion below.

## Two-language crypto model

E2EE is **per-protocol**, matching the "one relay bridges one protocol" architecture:

| Transport | E2EE scheme | Rationale |
|---|---|---|
| **mail** (jmapsmtp) | **PGP** (openpgp.js) | DeltaChat / chatmail interop ŌĆö non-negotiable |
| **ActivityPub** (jmapap) | **MLS** (target; see below) | fediverse-standard E2EE, interoperable |

These key models cannot be unified ŌĆö PGP is a stateless static key, MLS is a stateful per-device ratchet. What *is* unified is the surface: same `email` identity, same WebFinger/actor + WKD discovery, same cryptenv envelope/login. **PGP is never used over AP** ŌĆö a PGP-wrapped Note is unreadable to Mastodon/Emissary/other fediverse clients and would defeat federation. AP is either plaintext (fediverse interop) or MLS (standard E2EE). The send path skips PGP entirely for AP relays (`isApRelay()` in `context.ts` gates `encryptText` and `prefetchRecipientKey`).

## MLS-in-ActivityPub ŌĆö decision & design (decided 2026-07-06, timing TBD, not yet implemented)

**Decision:** bisetŌåöbiset AP E2EE uses **MLS**, per the SWICG spec *"Messaging Layer Security in ActivityPub"* + **RFC 9420**, via the **`ts-mls`** TypeScript library (in-browser). Reference implementation: **Emissary** (emissary.dev/e2ee ŌĆö Social Web Foundation project, Sovereign Tech Fund funded, same `ts-mls`, targeting mid-2026).

**Why MLS over PGP for AP:** forward secrecy + post-compromise security, real group semantics at scale (efficient add/remove rekey), and ŌĆö decisively ŌĆö it is the emerging fediverse standard, so encrypted messages still *federate* (interop with Emissary et al.). PGP over AP gives none of this and breaks interop.

**Services already provided by `go-jmapap` (reused, not rebuilt):**
- MLS **directory service** ŌåÉ WebFinger + Actor profiles (locate users / their key material).
- MLS **delivery service** ŌåÉ AP inbox/outbox (route opaque MLS ciphertext to clients).
- To add: **KeyPackage publishing** at the actor, and **routing of opaque MLS message types** to the client (server treats contents as opaque).

**Identity / device design (recommended; final ratification pending):**

| Axis | Decision |
|---|---|
| **Credential** | AP **actor URI** (resolved from `email` via WebFinger), `basic` credential. Unifies with the merge model (email = identity = actor). |
| **Trust** | KeyPackage fetched from the actor's own server over HTTPS ŌĆö the AP-MLS directory-service trust model (honest-but-curious server, no CA). Key transparency is future work. |
| **Multi-device** | **Per-device leaf** (true MLS): each browser has its own signing key + KeyPackages and is an independent group member. Private keys never leave the device. |
| **State storage** | MLS group state (ratchet/secrets) in **IndexedDB (device-local)**, *not* the synced vault. |
| **History** | **Not shared across a user's own devices** (MLS default; Emissary behaves the same). A new device joins fresh. |
| **Device management** | Publish a KeyPackage set (device list) at the actor; future **revoke** UI removes a leaf from all groups. |

**Rejected: shared identity key across devices.** MLS group state is a *mutating* ratchet; syncing it across devices via the vault means concurrent epoch mutation ŌåÆ corrupted group state, plus copying private keys destroys per-device PCS. It looks like the PGP one-key-per-email model but is a trap. Per-device leaves are how AP-MLS/Emissary work, so it is also the interoperable choice.

**Accepted UX costs of per-device:** adding a contact adds *all* of their devices; a freshly-added device cannot read prior history; revoking a device is a group operation.

**Near-term stance:** AP stays plaintext (fediverse interop); E2EE remains mail/PGP only until MLS work begins.
